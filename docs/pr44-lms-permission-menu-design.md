# PR44 LMS 권한/그룹 관리 메뉴 구조 설계안

## 0. 전제와 결론

### 운영 전제

- 체험단은 수강코드 없이 관리자가 이미 알고 있는 회원을 직접 선택해 코스 또는 패키지 권한을 부여한다.
- B2B는 수강코드, 즉 현재 바우처를 발급하고 기업 담당자에게 전달한 뒤 사용자가 프로필에서 직접 입력해 권한을 획득한다.
- 패키지, 코스, 수업, 레슨은 콘텐츠 편성/제작 영역이다.
- 체험단, 바우처/수강코드, 수강권, 사용자 권한은 콘텐츠를 누가, 언제까지, 어떤 경로로 이용할 수 있는지를 다루는 권한/혜택 운영 영역이다.

### 설계 결론

현재 `LMS 관리` 하위의 `그룹 관리`는 콘텐츠 관리와 권한 관리를 한 메뉴에 섞어 운영자가 목적을 판단하기 어렵다. 따라서 메뉴를 다음처럼 분리하는 것을 권장한다.

- `LMS 관리`: 콘텐츠 구조와 학습 단위 관리에 집중한다.
- `수강 권한 관리`: 체험단, B2B 수강코드, 직접 지급 수강권, 권한 이력/만료를 관리한다.
- `혜택/프로모션 관리`: 할인 쿠폰, 포인트처럼 결제/마케팅 혜택을 관리한다.

최종적으로 체험단은 별도 메뉴로 노출하되, 내부 데이터 모델은 `권한 캠페인` 또는 `운영 캠페인` 개념으로 일반화하는 것이 좋다. B2B 바우처는 `수강코드 관리`라는 운영자 언어로 재명명하고, 직접 지급은 `수강권 지급 관리`로 분리한다.

## 1. 관리자 메뉴 구조 제안

### 1.1 현재 구조의 문제

현재 구조는 다음과 같은 인지 비용을 만든다.

```text
LMS 관리
- 패키지 관리
- 코스 관리
- 수업 관리
- 레슨 관리
- 그룹 관리
```

문제는 `그룹 관리`가 나머지 메뉴와 성격이 다르다는 점이다.

| 메뉴 | 실제 성격 | 운영 질문 |
| --- | --- | --- |
| 패키지 관리 | 콘텐츠 상품/묶음 구성 | 무엇을 묶어서 제공할 것인가? |
| 코스 관리 | 콘텐츠 카탈로그/학습 과정 | 어떤 학습 과정을 만들 것인가? |
| 수업 관리 | 학습 단위/클래스 구성 | 어떤 수업을 노출할 것인가? |
| 레슨 관리 | 세부 학습 콘텐츠 | 어떤 레슨을 편성할 것인가? |
| 그룹 관리 | 기간제 접근 권한/대상자 관리 | 누가 언제까지 볼 수 있는가? |

운영자가 `권한 부여`, `수강코드 생성`, `만료 확인`, `기업별 사용 현황`을 처리하려 할 때 `LMS 관리 > 그룹 관리`로 들어가야 하는 구조는 메뉴명이 업무 목적을 충분히 설명하지 못한다.

### 1.2 권장 상위 구조

권장 메뉴는 다음과 같다.

```text
LMS 관리
- 패키지 관리
- 코스 관리
- 수업 관리
- 레슨 관리

수강 권한 관리
- 체험단 관리
- 수강코드 관리
- 수강권 지급 관리
- 권한 이력/만료 관리

혜택/프로모션 관리
- 쿠폰 관리
- 포인트 관리
```

### 1.3 분리 기준

| 구분 | 포함 메뉴 | 분리 기준 |
| --- | --- | --- |
| LMS 관리 | 패키지, 코스, 수업, 레슨 | 콘텐츠를 생성, 편성, 노출하는 업무 |
| 수강 권한 관리 | 체험단, 수강코드, 수강권, 권한 이력 | 특정 회원/조직/코드가 콘텐츠 접근권을 얻는 업무 |
| 혜택/프로모션 관리 | 쿠폰, 포인트 | 결제 할인, 적립, 마케팅 혜택 업무 |

이 구조의 핵심은 운영자가 `콘텐츠를 수정하려는지`, `접근 권한을 부여하려는지`, `결제 혜택을 관리하려는지`를 상위 메뉴에서 바로 구분할 수 있게 하는 것이다.

## 2. 체험단 관리 설계

### 2.1 추천 방향

체험단은 별도 메뉴인 `체험단 관리`로 제공하는 것을 추천한다.

다만 내부 구현 개념은 단순한 사용자 그룹이 아니라 다음 속성을 가진 `권한 캠페인`으로 보는 것이 좋다.

- 캠페인명
- 운영 목적: 체험단, 마케팅, 인플루언서, 제휴 리뷰 등
- 대상 콘텐츠: 코스 또는 패키지
- 대상 회원: 이미 가입한 회원 또는 관리자가 식별 가능한 회원
- 권한 시작일/종료일
- 자동 만료 여부
- 지급 상태, 회수 상태, 운영 로그

### 2.2 체험단을 그룹으로 관리하는 방식

#### 장점

- 여러 명에게 같은 기간과 같은 콘텐츠 권한을 일괄 부여하기 쉽다.
- CSV 업로드, 대상자 검증, 일괄 만료 같은 배치 운영에 적합하다.
- 캠페인 단위로 성과와 참여자 현황을 확인하기 좋다.
- 기존 `그룹 관리` 화면/데이터 개념을 일부 재사용할 수 있다.

#### 단점

- 운영자가 `그룹`이라는 단어만 보고 체험단 업무를 바로 떠올리기 어렵다.
- B2B, 내부 테스트, 체험단이 모두 섞이면 목록 필터에 의존하게 된다.
- 체험단 특화 정보, 예를 들어 리뷰 제출 여부, 체험 미션, 홍보 채널, 담당자 메모를 표현하기 애매하다.
- 체험단은 수강코드를 쓰지 않는다는 중요한 운영 차이가 그룹 안에서 흐려질 수 있다.

### 2.3 체험단 관리 별도 메뉴 방식

#### 장점

- 운영자가 `체험단 관리`라는 메뉴명만 보고 업무 목적을 즉시 이해한다.
- 체험단 전용 지표를 전면에 배치할 수 있다.
  - 진행중 체험단 수
  - 종료 예정 체험단 수
  - 참여 회원 수
  - 권한 지급 실패/확인 필요 건수
- 체험단 특화 필드를 자연스럽게 다룰 수 있다.
  - 캠페인 채널
  - 체험 목적
  - 리뷰/미션 상태
  - 담당 운영자
  - 체험 종료 후 전환 메모
- 수강코드가 없는 직접 권한 지급 플로우를 명확히 표현할 수 있다.

#### 단점

- 그룹 개념을 완전히 제거하면 B2B/내부 테스트와 공통 로직이 중복될 수 있다.
- 체험단, B2B, 내부 테스트를 통합 조회하는 관리자 화면이 별도로 필요할 수 있다.
- 초기 구현 시 메뉴와 도메인 용어를 재정리해야 한다.

### 2.4 최종 제안

운영자 UI는 `체험단 관리`로 분리하고, 내부 데이터/정책 레벨에서는 `권한 캠페인`의 한 유형으로 관리하는 하이브리드 구조를 추천한다.

```text
체험단 관리 화면
→ 내부적으로는 권한 캠페인 type = TRIAL/EXPERIENCE
→ 회원 직접 선택 또는 CSV 업로드
→ 코스/패키지 권한 부여
→ 종료일 기준 자동 만료
```

이렇게 하면 운영자는 쉬운 메뉴명을 사용하고, 시스템은 B2B/내부 테스트와 공통 권한 부여·만료 로직을 재사용할 수 있다.

## 3. 바우처/수강코드 관리 설계

### 3.1 용어 제안

현재 바우처가 B2B 수강코드 역할이라면 운영자 메뉴명은 `바우처 관리`보다 `수강코드 관리`가 더 적합하다.

- 운영자 관점: 기업에 전달할 `수강코드`를 만든다.
- 사용자 관점: 프로필에서 `수강코드`를 입력한다.
- 시스템 관점: 코드는 권한을 발급하는 토큰이다.

따라서 내부 엔티티명은 `voucher`를 유지하더라도, 화면 라벨과 메뉴명은 `수강코드`로 통일하는 것이 좋다.

### 3.2 B2B 수강코드 운영 흐름

```text
관리자
→ 수강코드 캠페인 생성
→ 기업/계약 정보 입력
→ 지급할 코스 또는 패키지 선택
→ 코드 정책 설정
→ 코드 생성
→ 기업 담당자에게 전달
→ 사용자가 프로필에서 코드 입력
→ 권한 발급
→ 사용/미사용/만료/회수 현황 모니터링
```

### 3.3 수강코드 관리 화면 구조

#### 목록 화면

목록 화면은 `코드 단건`보다 `수강코드 캠페인` 중심으로 보는 것이 좋다.

| 항목 | 설명 |
| --- | --- |
| 캠페인명 | 예: 노바테크 2026 Q2 임직원 영어 과정 |
| 기업/거래처 | B2B 담당 기업명 |
| 대상 콘텐츠 | 코스 또는 패키지 |
| 코드 유형 | 공용 코드, 개인별 코드, 일회성 코드 |
| 발급 수량 | 생성된 코드 수 |
| 사용 수량 | 사용 완료 코드 수 |
| 미사용 수량 | 아직 입력되지 않은 코드 수 |
| 만료 수량 | 만료된 코드 수 |
| 권한 기간 | 코드 사용 후 부여되는 수강 기간 |
| 코드 입력 가능 기간 | 사용자가 코드를 입력할 수 있는 기간 |
| 상태 | 준비중, 배포중, 종료 예정, 종료, 중지 |

#### 상세 화면 탭

```text
수강코드 캠페인 상세
- 기본 정보
- 코드 목록
- 사용 현황
- 대상 콘텐츠
- 기업 담당자/배포 메모
- 만료/중지 정책
- 운영 로그
```

### 3.4 생성 정책

생성 단계에서는 다음을 명확히 선택하게 한다.

| 정책 | 선택 예시 | 설계 이유 |
| --- | --- | --- |
| 코드 유형 | 공용 코드 / 개인별 코드 / 일회성 코드 | B2B 계약 형태가 다를 수 있음 |
| 사용 가능 횟수 | 1회, N회, 무제한 | 공용 코드 남용 방지 |
| 대상 콘텐츠 | 코스 또는 패키지 | 권한 발급 대상 명확화 |
| 권한 기간 | 고정 종료일 또는 사용일 기준 N일 | 기업 계약과 학습 시작일 정책 대응 |
| 입력 가능 기간 | 배포 시작일~입력 마감일 | 계약 기간 외 입력 방지 |
| 도메인 제한 | 특정 이메일 도메인 선택 가능 | B2B 외부 유출 방지 |
| 중복 사용 제한 | 회원당 1회, 동일 캠페인 중복 불가 | 재입력/중복 지급 방지 |

### 3.5 배포 관리

배포는 실제 시스템이 이메일을 보내지 않더라도 운영 기록이 남아야 한다.

권장 기능은 다음과 같다.

- 코드 CSV 다운로드
- 기업 담당자 정보 등록
- 배포일, 배포 방식, 전달 메모 기록
- 배포 상태 변경: 준비중 → 배포중 → 종료
- 유출/오발송 시 캠페인 중지 또는 특정 코드 폐기

### 3.6 사용 현황 관리

사용 현황은 기업 담당자 문의에 빠르게 답할 수 있도록 구성한다.

| 지표 | 설명 |
| --- | --- |
| 총 발급 | 생성된 코드 수 |
| 사용 완료 | 실제 회원이 입력해 권한이 발급된 수 |
| 미사용 | 아직 입력되지 않은 수 |
| 만료 | 입력 가능 기간이 지나 사용할 수 없는 수 |
| 사용률 | 사용 완료 / 총 발급 |
| 최근 사용 | 최근 코드 입력 일시 |
| 사용자 목록 | 사용 회원, 이메일, 권한 시작/종료일 |

### 3.7 만료/회수 정책

수강코드는 두 가지 만료가 분리되어야 한다.

| 만료 종류 | 의미 | 예시 |
| --- | --- | --- |
| 코드 입력 만료 | 사용자가 더 이상 코드를 입력할 수 없음 | 2026-08-31 이후 입력 불가 |
| 수강 권한 만료 | 이미 획득한 수강 권한이 종료됨 | 입력일로부터 90일 후 종료 |

이 둘을 분리하지 않으면 `코드는 만료됐지만 이미 사용한 회원의 수강권은 유지되는가?` 같은 운영 질문에 답하기 어렵다.

## 4. 수강권 지급 관리 설계

### 4.1 역할 정의

`수강권 지급 관리`는 관리자가 특정 회원에게 코스 또는 패키지를 직접 부여하는 메뉴다.

체험단과의 차이는 다음과 같다.

| 구분 | 체험단 관리 | 수강권 지급 관리 |
| --- | --- | --- |
| 지급 단위 | 캠페인/그룹 중심 | 회원 또는 소수 대상 중심 |
| 목적 | 마케팅, 체험, 리뷰, 제휴 | CS 보상, 운영 조정, VIP 지급, 수동 보정 |
| 대상자 | 여러 명일 가능성이 높음 | 단건 또는 소수 처리 가능성이 높음 |
| 화면 핵심 | 캠페인 진행 현황 | 지급/회수 이력과 사유 |
| 특화 정보 | 캠페인명, 채널, 종료 예정 | 지급 사유, 승인자, 티켓/주문 연결 |

### 4.2 권장 기능

#### 수강권 지급 생성

관리자는 다음 흐름으로 지급한다.

```text
회원 검색
→ 회원 선택
→ 지급 대상 선택: 코스 또는 패키지
→ 권한 기간 선택
→ 지급 사유 입력
→ 필요 시 주문/문의/내부 티켓 연결
→ 확인 후 지급
→ 회원 권한 이력에 기록
```

#### 필수 입력값

| 항목 | 설명 |
| --- | --- |
| 회원 | 이메일, 휴대폰, 회원 ID로 검색 |
| 지급 대상 | 코스 또는 패키지 |
| 권한 시작일 | 즉시 또는 예약 시작 |
| 권한 종료일 | 고정일 또는 N일 후 |
| 지급 사유 | CS 보상, 운영 지급, 오류 보정, VIP 제공 등 |
| 관리자 메모 | 내부 확인용 |
| 승인/검토 | 고가 패키지 또는 장기 권한일 경우 선택 적용 |

#### 목록 화면

`수강권 지급 관리` 목록은 지급 이력 중심으로 구성한다.

| 항목 | 설명 |
| --- | --- |
| 지급 ID | 수동 지급 고유 식별자 |
| 회원 | 이름, 이메일, 회원 ID |
| 지급 대상 | 코스/패키지명 |
| 지급 유형 | 직접 지급, CS 보상, 운영 보정 등 |
| 권한 기간 | 시작일~종료일 |
| 상태 | 예약, 활성, 만료, 회수 |
| 지급자 | 처리 관리자 |
| 지급일 | 처리 일시 |
| 사유 | 지급 사유 요약 |

### 4.3 회수/만료 관리

직접 지급된 수강권은 반드시 회수와 만료 이력을 남겨야 한다.

권장 상태는 다음과 같다.

```text
예약
→ 활성
→ 만료

활성
→ 수동 회수

예약
→ 취소
```

회수 시에는 다음 정보를 요구하는 것이 좋다.

- 회수 사유
- 회수 처리자
- 회수 일시
- 회원에게 노출 여부
- 관련 문의/주문/관리자 메모

| 메뉴 | 운영 목적 | 주요 사용자 행동 |
| --- | --- | --- |
| 체험단 관리 | 수강코드 없이 알려진 회원에게 기간제 권한 지급 | 캠페인 생성, 회원 선택/업로드, 권한 부여, 자동 만료 확인 |
| 수강코드 관리 | B2B 기업 대상 수강코드 생성/배포/사용 현황 관리 | 코드 생성, CSV 다운로드, 사용률 확인, 코드 만료/중지 |
| 수강권 지급 관리 | 특정 회원에게 코스/패키지 직접 지급 | 회원 검색, 지급 대상 선택, 기간 설정, 지급/회수 |
| 권한 이력/만료 관리 | 전체 권한의 상태와 만료 예정 건 모니터링 | 회원별 권한 조회, 만료 예정 확인, 회수/재지급 추적 |

### 5.3 `그룹 관리`의 처리 방향

`그룹 관리`는 최종 메뉴에서 그대로 유지하지 않는 것을 권장한다.

대신 다음 중 하나로 전환한다.

#### 권장안: 운영 메뉴로 분해

```text
기존 그룹 관리
→ 체험단 관리
→ 수강코드 관리의 B2B 캠페인
→ 내부 테스트 권한은 수강권 지급 관리 또는 권한 이력에서 처리
```

이 방식은 운영자가 실제 업무명으로 접근할 수 있어 가장 이해하기 쉽다.

#### 보조안: 고급 메뉴로 유지

공통 캠페인/그룹 엔티티를 운영해야 한다면 일반 운영자 메뉴에서는 숨기고, 관리자/개발 운영자용으로만 `권한 캠페인 관리`를 둘 수 있다.

```text
수강 권한 관리
- 체험단 관리
- 수강코드 관리
- 수강권 지급 관리
- 권한 이력/만료 관리
- 권한 캠페인 관리(고급/관리자 전용)
```

다만 일반 운영자에게는 `그룹 관리`라는 추상 메뉴명을 노출하지 않는 편이 좋다.

## 6. 권한 도메인 공통 모델 제안

구현 방향을 검토할 때 다음처럼 공통 모델을 잡으면 메뉴는 분리하면서도 로직 중복을 줄일 수 있다.

### 6.1 핵심 엔티티

| 엔티티 | 역할 |
| --- | --- |
| Entitlement | 회원이 특정 코스/패키지에 접근할 수 있는 실제 권한 |
| EntitlementGrant | 권한이 발급된 사건 또는 지급 이력 |
| EntitlementCampaign | 체험단/B2B처럼 여러 권한 발급을 묶는 운영 캠페인 |
| AccessCode | B2B 사용자가 입력하는 수강코드 |
| EntitlementAuditLog | 생성, 지급, 사용, 만료, 회수 이력 |

### 6.2 발급 경로 구분

| 발급 경로 | 설명 | 연결 메뉴 |
| --- | --- | --- |
| TRIAL_DIRECT | 체험단 직접 지급 | 체험단 관리 |
| B2B_ACCESS_CODE | 수강코드 입력 지급 | 수강코드 관리 |
| ADMIN_GRANT | 관리자 직접 지급 | 수강권 지급 관리 |
| ORDER_PURCHASE | 결제 구매로 지급 | 주문/결제 관리 또는 회원 상세 |
| INTERNAL_TEST | 내부 테스트 지급 | 수강권 지급 관리 또는 고급 캠페인 |

이렇게 구분하면 회원 상세에서 `이 회원이 왜 이 콘텐츠를 볼 수 있는지`를 명확하게 설명할 수 있다.

## 8. 단계적 전환안

### 8.1 1단계: 메뉴와 용어 정리

- `LMS 관리`에서 `그룹 관리`를 제거하거나 숨긴다.
- `수강 권한 관리` 상위 메뉴를 추가한다.
- 기존 `Vouchers` 메뉴는 `수강코드 관리`로 재명명한다.
- 기존 `그룹 관리`의 마케팅 체험단 성격 화면은 `체험단 관리`로 재해석한다.

### 8.2 2단계: 운영 플로우 분리

- 체험단: 회원 직접 선택/업로드 → 코스/패키지 권한 지급 → 자동 만료.
- B2B: 수강코드 캠페인 생성 → 코드 배포 → 사용자 입력 → 권한 지급.
- 직접 지급: 회원 상세 또는 수강권 지급 관리에서 관리자 지급.

### 8.3 3단계: 권한 이력 통합

- 모든 권한 발급 경로를 `권한 이력/만료 관리`에서 조회한다.
- 회원 상세에도 동일한 권한 출처와 만료 정보를 표시한다.
- 만료 예정, 지급 실패, 회수 필요 건을 운영 대시보드화한다.

## 9. 최종 권장안 요약

1. `그룹 관리`는 LMS 하위 메뉴로 두지 않는 것을 추천한다.
2. 콘텐츠 관리와 접근 권한 관리를 분리한다.
3. 체험단은 운영자 메뉴로는 `체험단 관리`를 제공하고, 내부 모델로는 권한 캠페인 유형으로 관리한다.
4. 바우처는 B2B 운영 언어에 맞춰 `수강코드 관리`로 재명명한다.
5. 수강권 직접 지급은 `수강권 지급 관리`로 별도 메뉴화한다.
6. 전체 권한 상태는 `권한 이력/만료 관리`에서 통합 조회한다.
7. 최종 메뉴는 다음 구조를 권장한다.

```text
LMS 관리
- 패키지 관리
- 코스 관리
- 수업 관리
- 레슨 관리

수강 권한 관리
- 체험단 관리
- 수강코드 관리
- 수강권 지급 관리
- 권한 이력/만료 관리
```
